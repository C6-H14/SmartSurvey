#### 3. 生成过程中的调试（Debugging）重点

在系统开发中，您必然会遇到以下两个常见痛点，建议提前做好应对：
* **PDF 双栏排版导致的换行混乱：**
  学术论文大多是双栏。直接提取文本时，容易把左栏第一行和右栏第一行连在一起读。
  * *调试方案：* 在 `core/pdf_parser.py` 中读取文本时，可以使用 PyMuPDF 的 `rect` 区域分割，或者使用 `page.get_text("blocks")` 按逻辑块提取，能有效保留双栏阅读顺序。
* **Markdown 语法的学术化输出：**
  大模型生成的对比表格或 LaTeX 公式经常会出现少一个 `|` 或少一个 `$` 的问题，导致前端渲染失败。
  * *调试方案：* 在 Prompt 中使用 **Few-Shot（少样本提示）**，给大模型提供一个完全正确的 LaTeX 矩阵和 Markdown 表格模板，并要求其必须输出指定格式。

### 第三部分：如何评估产出的文献综述论文质量？

为了确保产出的综述能拿到课程的优异成绩，在最终提交前，您需要按照以下 **4 个维度** 进行自我评估和修改：

| 评估维度 | 评分标准（自我核对） | 优化手段（如果质量不达标） |
| :--- | :--- | :--- |
| **事实准确性<br>(Factuality)** | • 论文提及的传感器、网络精度等数据是否与原 PDF 一致？<br>• 有无虚构、张冠李戴的现象？ | 对照自建系统输出的“对比矩阵”表格，手动打开原 PDF 搜索关键词，核实核心结论。 |
| **学术逻辑性<br>(Structure)** | • 综述是否摒弃了“流水账”式的罗列？<br>• 是否按照“感知-决策-交互预警”的技术脉络展开？ | 调整 `core/templates.py` 中的生成大纲 Prompt，要求大模型必须采用总-分-总结构。 |
| **图表深度<br>(Depth of Visuals)** | • 论文内是否包含一个维素丰富、对比详尽的**对比表格**？<br>• 能否清晰展示不同算法的优缺点？ | 检查自动生成的 LaTeX/Markdown 矩阵，手动加入“大研项目实地硬件约束（如 RealSense 帧率）”的对比项。 |
| **批判性与前瞻性<br>(Critical Thinking)** | • 论文是否有您独立思考的“研究缺口（Gaps）”？<br>• 是否结合了您的具体工程场景？ | **这部分必须人机协同：** 由 AI 帮您梳理出大致方向，您根据大研项目实地部署的难点（如环境光线变化、机械臂反光等真实痛点），人工撰写“未来面临的工程挑战”章节。 |
