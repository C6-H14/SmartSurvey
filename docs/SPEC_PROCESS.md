# SmartSurvey SPEC Process

本文档记录 SmartSurvey 在 `SPEC.md` 形成前后的 Brainstorming 过程。它不是功能规约本身，而是需求成型过程、工程取舍、思想交锋和反思记录。

对应主文档：`docs/SPEC.md`  
对应过程日志：`docs/Agent_log.md`

## 1. Brainstorming 总览

本轮 Brainstorming 的起点是 AI4SE 期末项目《SmartSurvey: 自动文献综述与系统决策软件》。项目被明确为非 harness 应用类项目，目标不是做一个简单 demo，而是形成一个可交付、可演示、可解释安全边界的自动文献综述软件。

讨论最初围绕四个核心要素展开：

1. 五个 INVEST 用户故事。
2. PDF 批量解析、清洗提取、学术矩阵抽取和综述生成。
3. 使用操作系统 keyring 进行 API Key 安全存储、录入、读取、更新和清除。
4. Docker 分发与公网部署后的用户 Key 安全配置。

在多轮讨论中，项目定位逐步从“自动生成综述草稿”升级为“可生成中文 LaTeX 学术论文全文手稿的证据约束型文献综述系统”。这一变化是本次 Brainstorming 的核心成果。

## 2. 关键节点与思想交锋

### 2.1 用户定位：从泛泛科研工具收敛到课程项目学生

最初，SmartSurvey 可能面向三类用户：课程项目学生、科研助理、工程或产品决策者。AI 建议先确定主用户，否则五个用户故事会分散到不同目标上，导致 SPEC 难以验收。

用户最终决定：

- 主用户：课程项目学生。
- 次用户：科研助理或实验室成员。

这一决策让项目优先服务用户自己的课程项目和后续项目参考需求，同时保留面向科研助理的批量整理能力。

思想交锋：

- AI 倾向于先用主用户约束范围，避免功能愿景过宽。
- 用户接受该建议，但强调项目首先是为自己服务，再扩展到更精妙、更强力的功能。

结果：

`SPEC.md` 中的五个用户故事围绕“导入 PDF -> 解析清洗 -> 学术矩阵 -> 证据审查 -> 综述全文生成”形成闭环。

### 2.2 防空气警告：从提示词约束升级为后端可验证规则

项目一开始就强调需要防范“空气警告、过度报警”。AI 提出定义：系统不能输出空泛风险判断，所有风险、局限和可靠性判断都必须绑定论文、页码或段落证据、触发原因和置信度。

用户接受该方向，并进一步把它推进为后端强制校验：

```python
evidence_quote in page_text[evidence_page]
```

思想交锋：

- AI 最初提出较宽的“论文名 + 页码/段落证据 + 触发原因 + 置信度/缺失字段”。
- 用户进一步收紧为更可测试的 `evidence_page + evidence_quote` containment 校验。

结果：

防幻觉不再只是 Prompt 规则，而成为 TDD 核心测试点和后端写入前的拦截机制。

### 2.3 输出物：从 Markdown/Docx 草稿升级为 LaTeX 全文手稿

AI 曾建议最终交付物可以从以下形式中选择：

- Markdown 综述草稿 + Markdown 对比矩阵。
- Word/Docx 综述草稿 + Markdown/CSV 矩阵。
- Markdown + Docx + CSV 全部导出。

用户推翻了这些选项，明确要求：

- 网页端：Markdown 实时预览综述和矩阵。
- 下载端：`survey_draft.tex`、`matrix_table.tex`、`references.bib`。
- 目标不是空洞草稿，而是可贴入 Overleaf 的中文学术论文全文手稿。

思想交锋：

- AI 的建议偏向第一版可控性和常见课程项目交付。
- 用户认为简单表格或空泛草稿无法说清学术问题，因此将目标提升为 full-text academic manuscript。

结果：

`SPEC.md` 将 `survey_draft.tex` 定义为 3000-5000 字左右的中文全文手稿，并要求六个完整章节。

### 2.4 LaTeX 结构：从 Chapter 改为 article-compatible section

用户提出章节结构时使用了 Chapter 1 到 Chapter 6 的描述。AI 指出，如果目标是可直接贴入 Overleaf 的论文全文，`article` 类默认没有 `\chapter{...}`，因此建议使用 `\section{...}`。

用户采纳该建议。

结果：

`SPEC.md` 明确使用 `\section{...}`，并建议使用 `ctexart` 或 XeLaTeX 兼容中文模板。

### 2.5 三个测试方向：从 3D YOLO 风格选题扩展到跨学科泛化

早期项目上下文中存在工业/机器人工位或自动化实验室空间异常检测方向，带有 3D/YOLO、视觉感知、空间决策、异常检测等工程色彩。用户随后明确提出最终需要在三个方向测试：

1. 基于深度学习的工业或自动化实验室场景空间异常检测。
2. Brauer-Manin 障碍相关研究。
3. 矩阵乘法理论复杂度上限。

这带来了重要冲突：如果 schema 只围绕 3D YOLO 或工业异常检测设计，那么它很难适配数学理论方向。

AI 建议采用“双层 schema”：

- 通用字段：所有论文都必须有，如题名、作者、研究问题、方法、创新、局限、证据页码和证据摘录。
- 领域字段：根据主题动态生成，如 `sensor`、`accuracy`、`latency`，或 `complexity_bound`、`algorithmic_technique`，或 `mathematical_object`、`obstruction_type`。

用户采纳该建议。

结果：

SmartSurvey 被定义为跨领域文献综述工具，而不是只服务 3D YOLO 或工业异常检测的窄工具。

### 2.6 PDF 解析：从章节级理想化抽取转向混合鲁棒策略

AI 曾提出两种可能：

- 强制识别多个章节。
- 或先提取全文文本和页码切片。

用户最终提出更稳健的混合方案：

- 对 `Abstract`、`Introduction`、`Conclusion`、`References` 做启发式正则识别。
- 对主干内容使用 fallback page slices，保证任何 PDF 至少能按页导入。
- 缺失章节标记为 `missing`，不编造。

思想交锋：

- 章节抽取有助于结构化，但容易受论文模板影响。
- 页码切片不够语义化，但稳定、可验证、便于证据绑定。

结果：

`SPEC.md` 采用“核心标准章节 + 兜底页码切片”的混合解析标准。

### 2.7 证据粒度：用户推翻段落编号方案

AI 曾建议第一版采用 B 方案：页码 + 段落编号 + 原文摘录。理由是段落级更严谨。

用户推翻该建议，理由明确：

- PDF 没有原生段落标签。
- 双栏、公式、表格会破坏几何分段。
- 段落编号会引入解析 Bug。
- 不符合 YAGNI 和 Complexity Reduction。

用户最终决定采用 A 方案升级版：

- `evidence_page`
- `evidence_quote`
- containment 校验

结果：

项目放弃第一版段落编号，将可验证性置于表面精细度之上。

### 2.8 Keyring 凭据模型：从 .env 便利性升级为安全考核点

早期项目已有 `.env` 相关配置和 API 切换说明，但 AI4SE 期末项目将凭据安全作为核心考核点。Brainstorming 中，项目安全模型被提升为：

- 本地运行优先使用 Python `keyring`。
- 首次录入 API Key 必须无回显。
- 支持读取、更新、清除。
- keyring 不可用时必须明确报错。
- 不静默降级为明文文件。
- 不把 Key 写入 Git、日志、UI、导出文件或 Docker 镜像。

思想交锋：

- `.env` 方便开发和切换模型，但安全性不足以作为最终项目亮点。
- keyring 更适合作为本地凭据安全模型，能体现工程安全意识。

结果：

`SPEC.md` 将 keyring 设为本地凭据安全的核心规约，并明确公网部署时不能让所有用户共享开发者 Key。

## 3. 三轮关键迭代节选

以下节选为过程性摘录，保留关键含义而非完整逐字转录。

### 迭代一：用户故事与目标用户

AI 提问：

> SmartSurvey 的主用户到底是谁？是课程项目学生、科研助理，还是工程/产品决策者？

用户决策：

> 主用户是课程项目学生，次要用户是科研助理。这个项目优先为我服务，我需要做其他项目时可以参考，在此基础上再考虑更精妙和强力的功能。

结果：

五个 INVEST 用户故事围绕课程项目学生主流程展开，但语言保留“科研人员或学生”，使次用户也能自然覆盖。

### 迭代二：输出物从草稿变为全文手稿

AI 建议：

> 从期末项目可控性看，可以选择 Markdown 草稿、Docx 草稿或 Markdown/Docx/CSV 组合，不建议第一版承诺全部导出。

用户决策：

> 我们采用另一个方案：网页端 Markdown 实时预览，文件下载端提供 `survey_draft.tex`、`matrix_table.tex`、`references.bib`。系统最终生成的不是空洞草稿，而是可以直接贴入 Overleaf 编译的学术论文全文手稿。

结果：

`SPEC.md` 将生成目标升级为中文 LaTeX full-text academic manuscript，并明确 3000-5000 字和六个章节。

### 迭代三：证据粒度与后端拦截

AI 建议：

> 第一版可以采用页码 + 段落编号 + 原文摘录，严谨性较强。

用户决策：

> 不采用段落编号。PDF 无原生段落标签，双栏、公式和表格会导致解析 Bug。采用页码 + 原文摘录，并运行 `evidence_quote in page_text[evidence_page]` 的 containment 校验。

结果：

系统把“防空气警告”落实为可测试的后端约束，而不是依靠大模型自觉。

### 迭代四：跨领域 schema

AI 建议：

> 三个测试方向差异很大，建议采用通用字段 + 领域字段的双层 schema。

用户决策：

> 完全同意。

结果：

SmartSurvey 避免被 3D YOLO 或工业异常检测方向锁死，可以同时支持数学理论类文献。

## 4. AI 建议的采纳与推翻

### 4.1 被采纳的建议

1. 主用户优先收敛  
   AI 建议先确定主用户，用户采纳。原因是课程项目学生主流程最能支撑 SPEC 的可验收性。

2. 防空气警告需要证据绑定  
   AI 建议把空泛警告约束为证据绑定规则，用户采纳并进一步强化。原因是该点能形成课程项目亮点。

3. 双层学术矩阵 schema  
   AI 建议通用字段和领域字段分层，用户采纳。原因是三个测试方向跨越工程视觉和数学理论，必须避免单领域硬编码。

4. LaTeX 使用 `\section` 而非 `\chapter`  
   AI 建议使用 article-compatible sectioning，用户采纳。原因是 Overleaf 上普通论文模板更适配 `article` 或 `ctexart`。

5. 中文论文输出需要 LaTeX 模板约束  
   AI 提醒中文输出涉及 `ctexart` 或 XeLaTeX 兼容性，已进入 SPEC。原因是中文 LaTeX 编译不是纯文本问题。

### 4.2 被用户推翻或修正的建议

1. 交付物不采用 Markdown/Docx/CSV 方案  
   AI 曾建议较轻量的交付物组合。用户推翻，要求 Markdown 前端预览 + LaTeX 全文导出。原因是项目需要体现学术深度，简单草稿不足以说明问题。

2. 证据绑定不采用段落编号  
   AI 曾建议页码 + 段落编号 + 原文摘录。用户推翻，原因是 PDF 段落重建复杂且脆弱，不符合 YAGNI 和复杂度控制。

3. 综述不被定义为空泛草稿  
   AI 早期使用“综述草稿”表达，用户修正为“学术论文全文手稿”。原因是最终产物必须接近可提交、可编辑、可放入 Overleaf 的完整论文形态。

4. `.env` 不足以作为最终凭据安全方案  
   早期项目已有 `.env` 便利切换 API 的背景，但用户和 AI4SE 要求共同推动其升级为 keyring 模型。原因是凭据安全是核心考核点，不能停留在开发便利性。

## 5. 关键工程决策清单

1. 主用户为课程项目学生，次用户为科研助理或实验室成员。
2. 五个用户故事为：PDF 导入解析、学术矩阵提取、证据约束审查、可视化矩阵对比、综述全文生成。
3. 输出语言为中文。
4. 网页端使用 Markdown 实时预览。
5. 下载端输出 `survey_draft.tex`、`matrix_table.tex`、`references.bib`。
6. `survey_draft.tex` 使用 `\section{...}`，建议基于 `ctexart`。
7. 三个测试方向覆盖工业空间异常检测、Brauer-Manin 障碍、矩阵乘法复杂度上限。
8. 学术矩阵采用通用字段 + 领域字段双层 schema。
9. PDF 解析采用核心章节启发式识别 + fallback page slices。
10. 证据绑定采用页码 + 原文摘录。
11. 后端执行 containment 校验，失败则拦截并提示空气警告。
12. 本地 API Key 使用 keyring，无回显录入，支持读取、更新和清除。
13. Docker 镜像不得包含 `.env`、keyring 数据、用户 PDF 或任何真实 API Key。
14. 公网部署必须让用户配置自己的 Key，不得共用开发者固定 Key。

## 6. 对本次 Brainstorming 的反思

### 6.1 做得好的地方

第一，讨论没有停留在“让大模型帮我总结论文”的泛泛层面，而是把幻觉、证据绑定、PDF 解析失败、凭据泄露、Docker 镜像污染等工程风险提前纳入 SPEC。

第二，用户多次主动推翻过度复杂或不够学术的方案。例如段落编号看似更精细，但实际会把系统拖入 PDF 版面解析泥潭；Docx/CSV 看似常见，但不足以表达 Overleaf 学术论文交付目标。

第三，三类测试方向设计得足够异构。工业空间异常检测偏工程系统，Brauer-Manin 障碍偏纯数学，矩阵乘法复杂度偏理论计算机科学。这迫使 SmartSurvey 不能靠硬编码领域模板取巧。

### 6.2 仍需警惕的风险

第一，full-text academic manuscript 是高目标。3000-5000 字中文手稿很容易变成模型堆砌内容，因此后续实现必须用矩阵和证据先约束结构，再生成正文。

第二，containment 校验很实用，但也会遇到 PDF 提取文本中的换行、连字符、空格和 ligature 问题。实现时需要考虑规范化匹配，例如压缩空白、处理断词，而不是只做最朴素的字符串包含。

第三，keyring 在本地单用户场景合理，但公网多用户部署不能简单复用 OS keyring。后续需要单独设计公网用户级加密凭据存储，否则安全模型会在部署阶段断裂。

第四，中文 LaTeX 输出需要模板、转义和 BibTeX 兼容处理。尤其是特殊字符、数学公式、引用 key 和 `booktabs` 表格列宽，都可能影响 Overleaf 编译。

### 6.3 本次讨论的核心收获

SmartSurvey 的竞争力不在于“调用大模型生成一段综述”，而在于把学术综述拆成可验证的工程流水线：

```text
PDF page slices
-> structured academic matrix
-> evidence containment validation
-> Markdown preview
-> Chinese LaTeX manuscript
```

这条链路让项目同时具备 AI 应用价值、软件工程可测试性和安全考核亮点。

## 7. 后续建议

下一阶段不应直接开始大规模编码，而应先把 `SPEC.md` 中的关键测试点转化为实现计划：

1. `pdf_parser.py` 的 page slices 与核心章节识别。
2. evidence containment 的单元测试。
3. keyring 凭据管理 CLI 或 UI 流程。
4. academic matrix 的 JSON schema。
5. LaTeX 导出模板与最小可编译样例。

这些任务应优先围绕“可验收、可演示、可测试”展开。

### 8.2 冷启动测试结果报告

- **测试执行人**：全新、无上下文的 Cline 智能体（首次接入本项目，无历史对话）。
- **测试上下文**：仅提供项目目录结构、任务描述，以及 `docs/SPEC.md` 与 `docs/PLAN.md` 两份文档。
- **执行任务**：PLAN.md 中的 Task 1（测试基建与依赖基线）与 Task 2（核心数据模型）。

#### 测试结论

SPEC 与 PLAN 的清晰度极高。Task 1 与 Task 2 盲跑 100% 成功：

- Task 1（7 个子步骤）：Red（测试不存在 → 预期错误）→ Green（所有测试一次性通过）→ Commit（`b069c9a`）。
- Task 2（5 个子步骤）：Red（`ModuleNotFoundError` 确认）→ Green（`2 passed`）→ Commit（`5291fc1`）。
- 全程未触发任何"暂停提问"事件，规约覆盖了所有实现所需的细节。

#### 发现的问题（缺陷记录）

- **平台特性限制**：Windows PowerShell 不支持 `&&` 命令链。在执行 `git add ... && git commit ...` 时抛出语法错误。
- **影响范围**：仅影响命令行执行方式，不涉及核心业务逻辑或规约设计。
- **严重程度**：低（本地环境差异，非设计缺陷）。

#### 修正决策

- 无需修改核心业务设计。
- 在后续的 Windows 终端任务执行中，Git 提交操作需分步执行：`git add <files>` 完成后独立执行 `git commit -m "..."`。
- 该约束已在 `docs/Agent_log.md` 的 Task 1.1 的"Lessons learned"中记录，供后续智能体参考。
